## 上下文

> 2026-03-22 实现校准说明：
> 当前实现已按新的基础设施约束做了收缩。短期认证状态不再持久化到 MySQL，而是进入 Redis / Django cache；`PlannerRunRecord` 已被移除，`/api/plans/` 兼容接口直接复用 `PlanningJob`；排队取任务优先走 Redis，不再默认依赖数据库轮询；`AnalyticsEvent` 已从 MySQL 移出，改为结构化日志 + cache/Redis 最近事件；数据库保留的主表已经统一为下划线命名，并且 `ChatSession` 不再重复保存完整消息列表。

当前“旅游规划 Agent”已经完成了以对话为主的旅行规划工作台：用户可以通过聊天或快捷表单触发 LangChain + LangGraph 工作流，查看执行轨迹、路线草案、预算、来源信息和澄清问题。但现有系统仍然停留在演示态：

- Django 仍使用本地 SQLite，聊天会话和规划运行状态实际由内存 store 持有。
- `TripPlannerService` 通过进程内 `Thread` 异步执行规划任务，服务重启后正在运行或已生成的状态都会丢失。
- 用户没有账号体系，无法拥有自己的历史会话、方案版本和可追溯资产。
- 前端只有单工作台，没有“我的会话”“我的方案”“登录态”“最近会话恢复”等产品级入口。
- 当前输出结构已经适合“路线草案”，但还没有成为“可保存、可继续修订、可长期使用”的用户资产。

本次变更的目标不是替换现有 LangChain + LangGraph 规划核心，而是在其外部补齐产品底座：MySQL 持久化、轻量账号体系、会话/方案资产模型、可靠任务执行和基础权限观测能力。

已知约束：

- 后端继续使用 Django，前端继续使用 Vue 3。
- 模型编排能力必须继续基于 LangChain + LangGraph，而不是退回到直接拼装模型 HTTP 接口。
- 当前环境已存在单轮 `/api/plans/` 和对话式 `/api/chat/sessions/*` 能力，新设计应优先复用它们的结果结构与工作流。
- 用户已经明确提供目标 MySQL 资源：主机 `110.40.207.231`，数据库 `trip_assistant`。实际账号密码必须通过环境变量配置，不能写死在仓库中。

主要利益相关者：

- 终端用户：希望把旅行规划当作可反复回来的个人工具，而不是一次性问答。
- 产品实现者：需要一个从游客体验到登录沉淀再到方案管理的完整闭环。
- 后端实现者：需要把内存状态和线程任务迁移为数据库与可恢复执行。
- 前端实现者：需要在现有工作台基础上补齐用户态、历史资产和页面路由。

## 目标 / 非目标

**目标：**

- 让用户能够通过 Web UI 完成“提出需求 -> 多轮澄清 -> 获得路线 -> 保存方案 -> 后续继续编辑”的完整闭环。
- 让登录用户拥有可追溯、可管理的会话和方案资产，刷新页面或服务重启后数据不丢失。
- 引入 MySQL 作为系统记录源，承载用户、会话、回合、方案、验证码和任务状态。
- 将当前进程内线程执行改造为数据库持久化任务队列与独立 worker，支持排队、运行、完成、失败和超时。
- 在现有工作台上增加登录态、会话列表、方案列表、最近会话恢复和基础权限控制。
- 为后续运营与质量优化保留可观测数据基础，包括会话量、回合量、方案保存量、模型/数据源耗时与游客转化路径。

**非目标：**

- 不在本期实现 OTA 预订、支付、库存管理或订单履约能力。
- 不实现多人实时协作、企业组织空间或复杂角色权限系统。
- 不在本期实现专业导航级逐转指令、地图拖拽排序或局部重算 UI。
- 不强制本期接入第三方 OAuth；首版优先做轻量验证码登录。
- 不在本期完成分享链接、收藏候选点位和 PDF 导出，这些保留给后续 P1/P2。

## 决策

### 决策 1：以 MySQL + Django ORM 作为系统记录源，当前规划结果采用“结构化字段 + JSON 快照”混合建模

后端将引入 MySQL 作为主存储，生产和联调环境通过环境变量指向 `110.40.207.231` 上的 `trip_assistant` 数据库；测试环境仍允许使用 SQLite 以保持自动化测试成本可控。数据模型上采用“核心关系字段 + JSON 快照”混合模式：

- 强关系字段：`User`、`ChatSession`、`ChatTurn`、`TripPlan`、`AuthCode`、`PlanningJob` 之间的外键与状态字段。
- 快照字段：已确认约束、规划请求、执行步骤、结果结构、数据源状态等保持为 JSON，以减少与现有 Pydantic schema 的双重维护成本。

建议的核心对象：

- `User`: 用户账号，支持邮箱登录，预留手机号字段。
- `AuthCode`: 验证码记录，保存目标地址、用途、过期时间、状态、请求 IP 和限流信息。
- `ChatSession`: 会话主记录，包含标题、所有者、当前状态、最新结果摘要、最近访问时间。
- `ChatTurn`: 每一轮输入与输出，保存用户消息、输入模式、任务状态、执行步骤、结果快照和错误。
- `TripPlan`: 方案版本，保存来源会话、来源回合、版本号、草案/最终状态、约束快照和结果快照。
- `PlanningJob`: 异步任务记录，保存任务类型、目标对象、状态、超时时间、开始/结束时间、错误信息和指标快照。
- `AnalyticsEvent`: 轻量事件表，记录会话创建、回合完成、方案保存、游客转注册等行为。

理由：

- 现有规划输出已经是结构化对象，但字段演进仍然较快，使用 JSON 快照可以降低 schema 联动成本。
- 仅靠内存对象无法支撑刷新恢复、历史资产和任务可靠性。
- 全量关系型拆解结果字段会显著增加实现复杂度，并拖慢迭代。

备选方案：

- 继续使用 SQLite：适合单机演示，不适合远程部署和多进程 worker。
- 完全规范化结果表结构：长期更利于分析，但对首版实现成本过高。
- 直接改用 PostgreSQL：能力更强，但当前用户已明确提供 MySQL 环境，本期应优先贴合现有基础设施。

### 决策 2：首版采用“邮箱验证码登录 + JWT + 游客令牌迁移”，不引入密码和第三方 OAuth

认证方案采用无密码登录：

- `POST /api/auth/codes/` 请求邮箱验证码。
- `POST /api/auth/login/verify/` 校验验证码并发放 `access_token` / `refresh_token`。
- `GET /api/auth/me/` 查询当前登录用户。
- `POST /api/auth/logout/` 使 refresh token 失效并清理前端状态。

游客态通过浏览器本地保存 `guest_token` 持有当前会话归属；当用户登录或注册时，验证接口接收可选 `guest_token`，后端把对应游客会话和方案迁移到该用户账号下。

理由：

- 对消费型工具而言，验证码登录比用户名/密码更轻，注册与登录可以合并成一个流程。
- 前端当前是独立 dev server 调用 Django API，JWT bearer token 比基于跨站 cookie 的 session 认证更容易落地。
- 游客先体验后登录，是这类工具的关键转化路径。

备选方案：

- 用户名/密码：实现传统，但增加记忆负担和密码重置成本。
- Django session cookie：服务端能力成熟，但跨端口开发和前后端解耦场景下调试复杂。
- 手机验证码：符合国内习惯，但短信通道和成本管理超出本期最小范围。数据模型仍预留后续扩展位。

### 决策 3：保留“聊天优先 + 表单辅助”交互，但统一映射到同一个持久化会话

当前聊天和表单已经共享规划核心，本次不改变这一原则，而是继续推进为“同一会话、同一资产体系”：

- 用户在聊天工作台发送自然语言，创建 `ChatTurn(input_mode=chat)`。
- 用户在快捷表单提交结构化条件，创建 `ChatTurn(input_mode=form)`。
- 两种输入都写入同一 `ChatSession`，共享同一个已确认约束快照与最新结果。
- 前端结果区、执行轨迹区和消息区都围绕当前活动会话联动展示。

登录用户另外获得：

- 会话列表页 / 会话侧栏
- 最近会话自动恢复
- 会话重命名、删除
- 从历史会话继续编辑

游客获得：

- 基于 `guest_token + localStorage` 的最近会话恢复
- 保存方案时被引导登录，并在成功登录后自动迁移当前会话

理由：

- 这与产品定位一致：会话是用户逐步收敛需求的主载体。
- 如果表单和聊天分裂为两套记录体系，资产沉淀会变得混乱。
- 已有前端工作台可以直接扩展，不必重新发明第二套产品形态。

备选方案：

- 继续将表单生成看作独立 run：实现简单，但不符合“持续编辑”的产品目标。
- 强制用户登录后才能体验：资产管理更纯粹，但不利于首屏转化。

### 决策 4：以“数据库队列 + Django worker”替代进程内线程，不在本期强制引入 Redis / Celery

当前 `TripPlannerService` 使用 `Thread` 异步执行 `workflow.run_context(...)`。本次改为数据库持久化任务模型：

- API 创建 `PlanningJob(status=queued)` 并关联 `ChatTurn` 或单轮 `PlannerRun`。
- 独立 worker 进程（例如 `python manage.py run_planning_worker`）轮询数据库中的待处理任务。
- worker 通过数据库锁领取任务，执行 LangGraph 工作流，并把步骤、状态、结果和错误写回 `PlanningJob`、`ChatTurn`、`ChatSession`。
- 超时由 `lease_expires_at` / `timeout_seconds` 控制，worker 启动时可回收僵尸任务并标记失败或重新排队。

该方案优先依赖 MySQL，不强制增加 Redis。对于单机或小规模部署，它已经足以满足“非线程、可恢复、可持久化”的 P0 目标。

理由：

- 最直接地解决“服务重启后内存任务丢失”的问题。
- 相比 Celery + Redis，部署和调试链路更短，更适合当前产品阶段。
- 现有规划链路同步可调用，只需要改造调度和结果回写层，而不是重写工作流本身。

备选方案：

- 继续使用 `Thread`：实现零成本，但不满足可靠性目标。
- Celery + Redis：生态成熟，但会额外引入新的基础设施、部署和运维复杂度。
- 立即上 Kafka / 专用任务系统：明显超出本期范围。

### 决策 5：保留 LangChain + LangGraph 规划核心，外层改为“持久化上下文装配 + 任务执行器”

规划能力继续由现有 `PlanningNodes`、`TripPlannerWorkflow` 和 `QwenPlannerClient` 驱动，不改为原始接口拼装。新系统只在进入工作流前增加两个步骤：

- 从 `ChatSession` / `ChatTurn` / `TripPlan` 读取上下文，组装 `PlanningContext`
- 把结果和执行轨迹回写到数据库快照

同时增加两个新的输入来源：

- “从历史会话继续编辑”：基于 `ChatSession` 最新约束与结果生成新回合
- “从历史方案继续修订”：创建一个新会话，并用 `TripPlan` 的约束快照和结果快照作为种子上下文

理由：

- 这能最大限度复用现有模型编排和结果结构。
- 现有系统已经验证了 LangChain + LangGraph 工作流的可行性，本期主要问题不是模型编排本身，而是产品化承载。

备选方案：

- 重新定义第二套“持久化版规划工作流”：会带来重复实现和测试面膨胀。

### 决策 6：前端引入 Vue Router 和“用户工作台壳层”，但继续使用组合式状态而不是立刻引入大型状态库

现有前端只有单页 `App.vue`。本次建议引入 Vue Router，形成以下页面/模块：

- `/`：首页 / 规划工作台
- `/sessions`：我的会话
- `/plans`：我的方案
- `/plans/:id`：方案详情
- `/profile`：个人中心基础页

共享壳层负责：

- 顶部产品标题“旅游规划 Agent”
- 登录入口 / 用户菜单
- 最近会话快捷入口
- 游客态提示

状态管理上继续采用 Vue 组合式函数拆分为：

- `useAuth`
- `useWorkspace`
- `useSessionList`
- `usePlanLibrary`

而不是本期立即切到 Pinia。这样可以把改动控制在必要范围内，并与当前代码风格保持一致。

理由：

- 多页面与历史资产查看已经超出单 `App.vue` 能力，路由是必要的。
- 当前状态规模虽增大，但尚未复杂到必须引入新的全局 store 框架。

备选方案：

- 继续单页条件渲染：短期可行，但很快会在登录、列表页和详情页上失控。
- 立即引入 Pinia：可用，但不是本期实现闭环的必要条件。

### 决策 7：安全与观测作为产品基础能力进入 P0，而不是留到“后面再补”

首版即纳入：

- 认证接口限流：验证码发送和登录校验使用 DRF throttle + `AuthCode` 频控。
- 数据隔离：所有会话、方案、任务查询都必须按 `user_id` 或 `guest_token` 过滤。
- 请求校验与统一错误结构：避免前端对错误类型完全无感。
- 基础事件采集：会话创建、回合完成、澄清次数、方案保存、游客转登录。
- 基础性能指标：模型调用耗时、数据源调用耗时、任务总耗时、失败原因。

观测数据优先写入数据库表和结构化日志，不在本期构建复杂 BI 面板。

理由：

- 用户资产一旦进入数据库，权限隔离就不是“后续增强”，而是上线前提。
- 任务可靠性没有观测，就无法定位模型、数据源和队列问题。
- 后续优化澄清体验与注册转化必须依赖最基本的漏斗数据。

备选方案：

- 只打日志，不做结构化事件：实现快，但无法形成可靠统计。
- 先不做限流：短期简单，但验证码和认证接口很快会成为风险入口。

## 风险 / 权衡

- [MySQL 作为主存储后，本地开发和 CI 环境更复杂] → 保留 SQLite 测试回退路径，并通过环境变量明确区分本地/联调/生产配置。
- [数据库队列 worker 的吞吐能力不如专用任务系统] → 作为 P0 足够；后续如果并发显著增长，再平滑迁移到 Celery/Redis。
- [验证码登录需要邮件通道] → 首版先抽象发送接口，允许联调环境使用控制台或 mock provider，避免阻塞整体主链路实现。
- [JSON 快照不利于复杂 BI 查询] → 核心资产和状态字段保持关系型；只把高变动规划结果保留为 JSON，后续再按分析需求拆分。
- [游客会话迁移存在归属判断复杂度] → 统一用签名 `guest_token` 标识游客资产，并在登录成功时一次性 claim 当前 token 关联的数据。
- [前端页面数增加后，交互复杂度上升] → 先保证“工作台 + 会话列表 + 方案列表 + 方案详情”四类主流路径，个人中心保持极简。

## 迁移计划

1. 增加数据库配置层，接入 MySQL 驱动与环境变量，保留测试环境 SQLite。
2. 引入 Django 认证相关 app、自定义用户模型或用户扩展模型、验证码模型和 JWT 鉴权链路。
3. 落地 `ChatSession`、`ChatTurn`、`TripPlan`、`PlanningJob`、`AnalyticsEvent` 等 ORM 模型，并提供迁移脚本。
4. 将现有内存 `ChatSessionStore` / `PlannerRunStore` 替换为基于 ORM 的 repository 层。
5. 实现数据库队列 worker，完成从 API 入队到结果回写的链路，移除进程内线程执行。
6. 扩展前端为路由化工作台，补齐登录弹窗/页面、会话列表、方案列表与详情。
7. 增加游客态恢复、登录迁移、方案保存与继续修订链路。
8. 补齐权限校验、限流、基础事件与指标采集。
9. 在联调环境使用远程 MySQL `trip_assistant` 验证迁移、恢复和权限隔离。

回滚策略：

- 若 MySQL 集成出现阻断，可临时保留 SQLite 开发配置，但不发布“资产沉淀”功能。
- 若数据库队列 worker 不稳定，可暂时保留同步执行的只读调试入口，但主产品接口不再依赖线程。
- 若登录流程未准备好，可先保留游客工作台和本地恢复能力，再逐步打开方案保存与历史入口。

## 开放问题

- 首版验证码通道最终使用邮箱还是同时支持邮箱/手机号；如果只支持邮箱，产品文案需要同步收敛。
- 是否在 P0 就实现 `SavedPlace` 收藏能力，还是把它明确延后到 P1，避免拖慢账号 + 会话 + 方案主链路。
- 数据库队列 worker 是否需要“失败自动重试 1 次”作为默认策略，还是只做失败落库与人工重放。
- 方案详情页是否允许直接在页面中发起“继续修订”，还是统一跳回工作台并打开新会话。
