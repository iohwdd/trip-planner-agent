## ADDED Requirements

### 需求:用户必须能够把当前结果保存为草案或最终方案
系统必须允许用户将当前会话中的路线结果保存为旅行方案，并区分“草案”和“最终方案”两种状态。

#### 场景:用户保存当前路线草案
- **当** 用户在当前会话中点击“保存方案”并选择草案
- **那么** 系统必须创建一个新的方案记录
- **那么** 该记录必须保存当前约束快照、结果快照和保存时间

#### 场景:用户确认最终方案
- **当** 用户把某个可用路线保存为最终方案
- **那么** 系统必须将该版本标记为最终方案
- **那么** 方案列表和详情中必须能明确区分其状态

### 需求:一个会话必须支持多个方案版本
系统必须允许同一会话随着多轮修订产生多个方案版本，并为每个版本保存独立的版本号和来源回合。

#### 场景:用户连续保存两次修订结果
- **当** 用户在同一会话中先保存一个草案，又在后续修订后再次保存
- **那么** 系统必须创建新的方案版本，而不是覆盖旧版本
- **那么** 每个版本都必须能追溯到对应的来源回合和生成时间

### 需求:系统必须提供方案列表与方案详情
系统必须允许登录用户查看自己的历史方案列表，并查看任一方案的详情信息。

#### 场景:用户查看历史方案
- **当** 登录用户进入“我的方案”列表并选择一个方案
- **那么** 系统必须展示该方案的状态、版本号、生成时间和核心约束摘要
- **那么** 用户进入详情页后必须能够查看保存时的结果快照

### 需求:用户必须能够从历史方案继续修订
系统必须允许用户基于历史方案重新开始修订，而不是要求用户把旧方案内容手动复制回聊天输入框。

#### 场景:用户基于旧草案继续规划
- **当** 用户在方案详情页选择“继续修订”
- **那么** 系统必须创建或打开一个可继续编辑的会话上下文
- **那么** 新会话必须继承该方案保存时的约束快照或结果快照作为起点

### 需求:用户必须能够删除自己的方案且不得访问他人方案
系统必须允许用户删除自己的历史方案，并保证方案详情、列表和删除操作都遵循用户数据隔离。

#### 场景:用户删除自己的旧方案
- **当** 登录用户删除一个属于自己的历史方案
- **那么** 系统必须移除该方案在其个人列表中的可见性
- **那么** 该操作不得影响其他方案版本或其他用户数据

#### 场景:用户尝试访问其他人的方案
- **当** 用户请求一个不属于自己的方案详情或删除接口
- **那么** 系统必须拒绝访问
- **那么** 系统不得返回该方案的快照内容
