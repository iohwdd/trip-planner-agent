## 为什么

当前旅游规划 Agent 已经具备对话式规划、路线草案和执行轨迹展示能力，但它仍然更像一次性演示系统：后端依赖内存存储和进程内线程，用户没有账号体系，刷新或服务重启后会话与方案无法稳定保留。现在需要把系统升级成一个面向个人用户的可持续使用产品，让用户能够通过多轮对话沉淀自己的旅行会话和方案资产，并以 MySQL 持久化作为基础设施起点。

## 变更内容

- 引入基于 MySQL 的持久化数据层，替代当前仅依赖内存和本地 SQLite 的临时状态承载方式，用于保存用户、会话、回合、方案版本、验证码记录和任务状态。
- 新增轻量账号体系，支持注册、登录、退出、当前登录态查询，以及游客体验后迁移当前会话到登录账号。
- 将聊天工作台升级为可管理资产入口，支持登录用户查看历史会话、继续编辑、重命名、删除，并在刷新页面后恢复最近会话。
- 新增旅行方案管理能力，允许用户把当前路线保存为草案或最终方案，并保留会话来源、约束快照、结果快照和版本号。
- 将快捷表单继续作为次级入口，但要求其提交结果进入同一会话和同一资产体系，而不是独立的一次性流程。
- 改造当前规划执行机制，去除进程内线程依赖，改为可持久化的任务队列与执行器，支持排队、运行、完成、失败、超时和后续扩展重试/取消。
- 补齐基础权限控制、数据隔离、限流、错误记录和基础观测能力，使系统具备产品级最小可用性。

## 功能 (Capabilities)

### 新增功能
- `account-authentication`: 提供游客态、注册登录、登录态查询、退出登录、验证码验证与游客会话迁移能力。
- `trip-session-management`: 将旅行对话会话持久化为用户资产，支持历史查看、继续编辑、重命名、删除、最近会话恢复和聊天/表单统一入口。
- `trip-plan-management`: 支持将当前路线结果保存为草案或最终方案，保留版本号、来源会话、约束快照、结果快照，并支持方案列表、详情、删除和从历史方案继续修订。
- `planning-job-reliability`: 提供基于数据库的任务持久化、排队执行、状态跟踪、超时与失败记录，以及模型/数据源基础指标采集能力。

## 影响

- 将影响 `./backend/trip_planner_backend/settings.py`、`./backend/planner/api`、`./backend/planner/services` 和新增 Django ORM 模型与迁移，数据库配置将从默认本地 SQLite 扩展为面向 MySQL 的运行配置。
- 将影响当前聊天会话和规划运行机制，因为 `ChatSessionStore`、`PlannerRunStore` 与 `TripPlannerService` 需要从内存实现切换到数据库持久化与异步任务执行。
- 将影响前端 `./frotend/src` 的信息架构，需要在现有工作台之外新增登录入口、用户菜单、会话列表、方案列表、方案详情或侧栏管理能力，并处理游客态与登录态切换。
- 将影响 API 契约与权限控制，新增认证接口、用户资产接口、方案管理接口和任务查询接口，同时要求用户只能访问自己的数据。
- 将新增或调整运行依赖，至少包括 MySQL 驱动、认证验证码能力、任务执行组件以及基础限流/观测配置。
