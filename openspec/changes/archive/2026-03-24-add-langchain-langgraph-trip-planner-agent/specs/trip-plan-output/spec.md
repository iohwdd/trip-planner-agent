## 新增需求

### 需求:输出必须包含旅行概览与关键假设
系统必须为每份旅行方案输出旅行概览，并明确列出生成方案时使用的关键假设、限制条件或待用户确认的信息。

#### 场景:生成方案时附带假设说明
- **当** 系统成功生成一份旅行方案
- **那么** 输出结果必须包含行程概览以及假设、限制或待确认事项列表

### 需求:输出必须保持前端可渲染的结构化格式
系统必须以稳定的结构化字段返回旅行规划结果，使前端能够直接按模块渲染，而不是依赖自由文本解析。

#### 场景:前端根据固定字段渲染结果
- **当** 前端接收到一次成功的旅行规划响应
- **那么** 响应结果必须包含稳定字段，以支持前端按模块展示结果内容

### 需求:输出必须包含来源信息与时间信息
系统必须为基于外部数据源生成的推荐项输出来源名称、获取时间或刷新时间，以便用户判断结果的新鲜度和可信度。

#### 场景:推荐项附带来源与时间
- **当** 系统返回包含景点、美食或酒店推荐的旅行方案
- **那么** 输出结果必须包含对应推荐项的来源信息和时间信息

### 需求:输出必须覆盖逐日行程安排
系统必须为旅行天数内的每一天提供结构化行程安排，包含当天主题、推荐活动、建议节奏或时间段说明。

#### 场景:三天旅行生成三天安排
- **当** 用户请求一个三天的旅行方案
- **那么** 输出结果必须包含覆盖三天的逐日行程安排，且每天都有独立内容

### 需求:输出必须包含预算拆解与适配说明
系统必须给出预算拆解，说明住宿、交通、餐饮和活动等主要成本项的估算分配，并指出方案是否满足用户预算约束。

#### 场景:生成预算分配结果
- **当** 系统为带预算约束的请求生成旅行方案
- **那么** 输出结果必须包含主要成本项的预算分配，并说明总预算是否在用户限制内

### 需求:输出必须包含交通与住宿建议
系统必须基于目的地、天数和偏好提供交通方式与住宿策略建议，且这些建议必须与整体行程保持一致。

#### 场景:提供与行程一致的配套建议
- **当** 系统生成完整旅行方案
- **那么** 输出结果必须包含与逐日行程匹配的交通和住宿建议

### 需求:输出必须体现外部数据缺失或降级情况
当部分实时数据未能获得或存在降级处理时，系统必须在输出结果中明确说明缺失来源、影响范围和建议的人工确认项。

#### 场景:存在部分数据源降级
- **当** 系统在规划过程中未能成功获取全部外部数据
- **那么** 输出结果必须明确列出缺失来源、受影响模块和需用户确认的内容

### 需求:输出必须包含个性化建议与风险提示
系统必须根据用户偏好提供个性化建议，并提示可能影响行程执行的限制、风险或需要二次确认的事项。

#### 场景:存在高强度行程时给出提醒
- **当** 方案包含较高活动密度、较长通勤距离或预算边界较紧的安排
- **那么** 输出结果必须给出明确的风险提示或调整建议
