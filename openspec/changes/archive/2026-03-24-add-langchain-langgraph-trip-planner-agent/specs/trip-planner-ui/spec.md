## 新增需求

### 需求:界面必须收集旅行规划核心输入
前端界面必须允许用户输入预算、天数、目的地和出行偏好等核心规划条件，并在提交前展示清晰的字段含义。

#### 场景:用户填写规划表单
- **当** 用户访问旅行规划页面
- **那么** 系统必须展示可编辑的输入表单，且表单至少包含预算、天数、目的地和偏好字段

### 需求:界面必须展示旅行规划结果
当前端收到成功响应时，界面必须以模块化方式展示旅行概览、逐日行程、预算拆解、交通建议、住宿建议和风险提示。

#### 场景:成功返回结果时渲染方案模块
- **当** 后端返回成功的旅行规划结果
- **那么** 前端必须按结构化模块展示主要内容，而不是将结果压缩为单段原始文本

### 需求:界面必须展示 Agent 执行过程
前端界面必须以专门设计的可视化区域展示 Agent 执行过程，包括当前阶段、数据源调用状态和关键状态消息。

#### 场景:规划执行中展示过程反馈
- **当** 用户提交旅行规划请求且系统正在执行多步骤工作流
- **那么** 前端必须展示 Agent 当前执行阶段、已完成步骤和外部数据源调用状态

### 需求:界面必须处理加载态和失败态
前端界面必须在请求处理中展示加载反馈，并在请求失败或结果不可用时展示可理解的错误信息和下一步提示。

#### 场景:请求处理中显示反馈
- **当** 用户提交旅行规划请求且后端仍在处理中
- **那么** 前端必须展示明确的加载状态，避免用户误判系统无响应

### 需求:界面必须展示来源与可信度信息
前端界面必须为景点、美食和酒店等推荐项展示数据来源、更新时间或降级说明，帮助用户理解结果依据。

#### 场景:展示推荐项来源标签
- **当** 前端渲染包含外部数据来源的旅行规划结果
- **那么** 页面必须为相关推荐项展示来源信息、时间信息或降级提示

### 需求:界面必须采用蓝色系视觉风格
前端界面必须以蓝色系为主视觉，且表单、执行过程区块、结果区块、按钮和状态反馈保持统一的蓝色主题体系。

#### 场景:页面展示统一蓝色主题
- **当** 用户访问旅行规划页面并完成一次规划流程
- **那么** 页面中的主要交互元素和结果模块必须体现统一的蓝色系主题，而不是默认或混杂配色

### 需求:界面必须兼顾桌面端与移动端可用性
前端界面必须在桌面端和移动端都保持可读、可滚动和可操作，保证用户可以完成一次完整规划流程。

#### 场景:移动端查看规划结果
- **当** 用户在移动端设备上提交并查看旅行规划结果
- **那么** 页面必须保持关键信息可读，且表单与结果区块能够顺畅滚动与操作
