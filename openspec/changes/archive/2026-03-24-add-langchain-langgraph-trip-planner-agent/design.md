## 上下文

该仓库目前仍处于空白起步阶段，只有 OpenSpec 结构，没有既有实现可以复用。本次变更不再是一个只靠模型内部知识生成方案的最小可用原型，而是一个前后端一体的旅行规划系统，目标是基于实时外部数据尽可能生成真实、全面、可靠的旅行方案。

后端将放在 `./backend`，使用 Django 作为 Web 框架，LangChain 负责模型调用与结构化输出，LangGraph 负责多步骤状态流转、数据检索、校验、重试与兜底逻辑。前端将放在 `./frotend`，使用 Vue 3 构建交互界面，不仅展示最终旅行方案，还需要展示 Agent 的执行过程、数据源检索状态和中间步骤反馈。

模型接入方面，首版改为使用阿里云百炼千问 `qwen3.5-plus`。数据源方面，首版改为接入高德地图 POI 搜索与百度地图地点检索，用于支撑景点、美食和住宿候选信息。大众点评与携程的公开接入路径更偏合作或企业场景，不再作为首版必需依赖。由于这些外部接口仍然存在鉴权、配额、延迟和可用性限制，系统必须通过统一适配层、缓存策略、失败降级和来源标注来保证结果的可解释性和可用性。

约束如下：
- 后端必须使用 Django，并通过 HTTP API 对前端暴露旅行规划能力。
- 旅行方案必须尽可能依赖实时或近实时外部数据，而不是只基于模型常识生成。
- 结果必须是稳定的结构化输出，且要包含来源信息、抓取时间或刷新时间等可追踪字段。
- 当前用户指定前端目录为 `./frotend`，设计将按该目录组织前端代码。
- 前端整体视觉必须采用蓝色系，并为 Agent 执行过程和最终结果提供专门 UI，而不是普通表单页。
- 当输入缺失、外部数据不足或第三方接口失败时，系统必须给出明确的澄清项、降级说明或假设说明。

## 目标 / 非目标

**目标：**
- 在 `./backend` 中提供一个基于 Django 的旅行规划服务，接收预算、天数、目的地、出行偏好等输入。
- 使用 LangGraph 定义包含输入标准化、实时数据检索、候选组合、方案生成、结果校验和兜底处理的工作流。
- 通过千问 `qwen3.5-plus` 结合高德地图与百度地图数据生成更接近真实世界的旅行方案。
- 在 `./frotend` 中提供 Vue 3 界面，用于填写旅行条件、查看 Agent 执行进度、浏览外部数据来源和阅读最终方案。
- 使用蓝色系 UI 建立可信且有层次的产品观感，并保证桌面端与移动端都能完成完整流程。

**非目标：**
- 不实现真实下单、支付、库存锁定、订单确认或售后流程。
- 不覆盖多用户协同编辑、行程版本管理、登录系统或长期记忆。
- 不在首版实现全球范围的多供应商动态比价引擎。
- 不保证所有外部数据源在任何时间都完全可用，但必须在不可用时提供清晰降级结果。
- 不在首版接入除千问 `qwen3.5-plus` 之外的多模型自动路由机制。

## 决策

### 决策 1：后端使用 Django 作为 Web 框架，代码位于 `./backend`

后端统一使用 Django 提供 API、配置管理和服务边界，Agent 工作流运行在 Django 应用内部的服务层。

建议分层：
- Django project/settings：环境配置、密钥管理、日志和路由
- Django app/api：旅行规划接口、健康检查和错误响应
- service/graph：LangGraph workflow 与节点实现
- integrations：千问、高德、百度地图等第三方适配器
- domain/schema：请求、响应、状态和标准化数据结构

原因：
- Django 对配置管理、项目组织和部署习惯更稳定，适合作为长期后端骨架。
- 将 Web 层、Graph 层和第三方适配层拆开，后续维护成本更低。
- 用户已经明确指定 Django，因此应避免引入其他主框架。

备选方案：
- Flask/FastAPI：更轻量，但不符合当前明确技术约束。
- 在单文件脚本中直接运行 LangGraph：实现快，但后续无法承载更复杂的集成。

### 决策 2：通过独立模型适配层封装千问 `qwen3.5-plus`

后端不在业务节点中散落模型调用，而是通过独立适配层封装千问 `qwen3.5-plus` 的接入。

适配层职责：
- 读取模型配置和密钥
- 创建 LangChain 可用的模型客户端
- 统一处理超时、异常映射、重试和默认参数

原因：
- 模型供应商耦合必须被隔离，否则状态图节点将难以测试和替换。
- 统一异常处理可以让前端得到稳定、可渲染的错误结果。

备选方案：
- 在 LangGraph 节点内部直接调用供应商 SDK：开发更快，但维护和测试代价更高。

### 决策 3：通过多源数据适配层接入高德与百度地图

系统通过标准化适配层接入外部数据源，并将结果统一映射为内部候选实体。

建议适配对象：
- 高德地图 POI：景点、地标、地理位置与基础营业信息
- 百度地图地点检索：餐厅、美食候选、商圈与酒店候选
- 酒店候选：首版通过地图地点检索获取位置型住宿候选，并在结果中标注其并非 OTA 实时库存或实时报价

适配层职责：
- 处理鉴权、请求参数组装、响应解析与限流保护
- 将不同供应商的字段映射为统一的 `poi`, `food_place`, `hotel_option` 等内部模型
- 记录来源、获取时间和可用性状态

原因：
- 高德与百度地图都提供公开的开发者接入路径，更适合作为首版个人开发者方案。
- 多源数据是“真实、实时、可靠”要求的核心，没有独立适配层就无法稳定扩展。
- 统一内部模型可以降低后续规划节点的复杂度，并允许后续再引入 OTA 或本地生活商业合作接口。

备选方案：
- 先只用单一数据源：实现更简单，但无法满足景点、美食、住宿的全面性要求。
- 继续保留大众点评或携程占位适配器：名义上更全面，但会让首版依赖不可验证的企业合作前提。
- 直接把原始第三方响应交给 Graph：开发快，但强耦合且难以测试。

### 决策 4：采用“检索增强 + 状态图编排”的旅行规划流程

Agent 将由多个图节点组成，而不是依赖一次性大提示词直接生成最终结果。

建议的状态节点：
- `normalize_input`：解析并标准化预算、天数、目的地、偏好与约束
- `fetch_poi_candidates`：调用高德地图获取景点和地理候选
- `fetch_food_candidates`：调用百度地图地点检索获取餐饮与商圈候选
- `fetch_hotel_candidates`：调用百度地图地点检索获取酒店候选，并标注其属于地图候选而非 OTA 预订结果
- `validate_live_data`：过滤空结果、重复项、过期数据和明显不可信数据
- `compose_trip_context`：将用户约束与实时候选整合为规划上下文
- `plan_trip`：调用 `qwen3.5-plus` 生成逐日行程与推荐组合
- `verify_plan_consistency`：校验预算、时间跨度、住宿与交通逻辑一致性
- `assemble_output`：整理为稳定的结构化响应
- `fallback_or_clarify`：当输入不足、外部数据缺失或生成失败时输出澄清请求或降级方案

原因：
- 旅行规划不仅是内容生成，更是数据收集、筛选、约束满足与结果解释的过程。
- 显式状态便于调试单个数据源问题，也便于在前端展示执行进度。
- 将“检索”和“生成”分离，可以减少模型凭空编造细节。

备选方案：
- 单轮 prompt 直接规划：输出快，但真实性和可解释性不足。
- 多 Agent 协作：灵活，但首版复杂度过高且调试成本大。

### 决策 5：输出采用固定 schema，并附带来源与时效信息

旅行方案输出将定义稳定字段，由后端返回结构化 JSON，前端按模块渲染，并展示来源与时间信息。

建议输出结构包括：
- `trip_summary`
- `assumptions`
- `execution_summary`
- `daily_itinerary`
- `budget_breakdown`
- `transportation`
- `accommodation`
- `attractions`
- `food_recommendations`
- `recommendations`
- `warnings`
- `source_references`
- `generated_at`

原因：
- 稳定 schema 便于测试、前端渲染和后续 API 演进。
- 来源和时间信息是“可靠性”要求的基本组成部分，不能只体现在日志里。

备选方案：
- 只输出自然语言 Markdown：可读性高，但无法稳健渲染执行过程和来源信息。
- 前端自行解析文本：实现快，但易碎且不可控。

### 决策 6：对外部数据失败采用部分降级而非整体失败

当某个供应商失败时，系统优先保留其余可用结果并显式告知缺失来源，而不是整个规划流程直接失败。

降级策略：
- 单个来源失败时，继续生成方案，但在结果中注明哪些推荐缺少实时验证
- 所有关键来源均失败时，返回可恢复的失败结果，并建议用户重试或缩小查询范围
- 对频繁查询的基础地理信息使用短时缓存，以减少限流和重复调用

原因：
- 真实世界 API 不稳定是常态，整体失败会显著降低可用性。
- 用户更需要一个带不确定性说明的可读结果，而不是简单报错。

备选方案：
- 任一数据源失败即终止：一致性高，但可用性过低。
- 完全忽略失败来源：用户无法判断结果可信度。

### 决策 7：前端使用蓝色系、双层结果视图与执行过程时间线

前端不仅展示结果，还要把 Agent 的执行过程可视化，帮助用户理解系统如何检索和生成方案。

建议 UI 结构：
- 顶部或左侧为输入表单区，突出预算、天数、目的地和偏好
- 中间或右侧为执行过程区，展示当前阶段、已调用的数据源和状态消息
- 结果区以蓝色系卡片群展示每日安排、住宿、美食、预算与风险提示
- 数据源引用区展示推荐项的来源和更新时间

设计要求：
- 统一蓝色主题变量，避免默认样式拼接出的普通后台感
- 执行过程区需要专门的 stepper、timeline 或 event stream 设计
- 加载态、错误态、部分降级态都必须纳入统一视觉系统
- 在移动端保持输入、执行过程和结果区块的层次感与可滚动性

原因：
- 实时规划流程天然具有多个步骤，不展示过程会削弱产品可信度。
- 数据源和执行过程可视化能够支撑“真实、实时、可靠”的产品感知。

备选方案：
- 只展示最终结果：实现更快，但用户难以理解系统依据与等待进度。
- 使用默认组件库布局：成本低，但很容易做成普通表单页。

## 风险 / 权衡

- [第三方接口权限、配额或合作门槛不确定] → 首版优先选择可公开申请的地图开放平台，并在配置和文档中明确凭证前置条件。
- [多源数据返回格式不一致] → 建立统一内部 schema，并在接入层做字段映射与去重。
- [实时接口延迟较高，影响用户等待体验] → 在前端展示执行过程，在后端增加超时、重试和短时缓存。
- [供应商部分不可用时结果质量下降] → 使用部分降级策略，并在响应中展示缺失来源和置信说明。
- [模型输出仍可能与实时数据不一致] → 增加 `verify_plan_consistency` 节点，对预算、住宿、地点和时间安排做二次校验。
- [前后端同时起步且 UI 要求较高] → 首版聚焦单次规划请求、执行过程展示和结果阅读，不扩展到账户体系或订单系统。
- [当前前端目录名为 `frotend`] → 设计按该路径执行，若后续确认是拼写修正需求，再单独调整目录。

## 迁移计划

这是一个新增能力，无需对既有功能做数据迁移。

建议的落地顺序：
1. 在 `./backend` 初始化 Django 项目、依赖管理、配置模块和基础路由。
2. 建立旅行规划输入输出模型、千问 `qwen3.5-plus` 适配层和基于高德/百度的第三方数据接入适配层。
3. 实现 LangGraph 工作流，包括实时数据检索、校验、生成、验证与降级处理。
4. 暴露 Django HTTP API，并定义稳定的请求响应契约。
5. 在 `./frotend` 初始化 Vue 3 项目，建立蓝色系主题和页面骨架。
6. 实现输入表单、执行过程可视化、结果展示和来源信息展示。
7. 增加前后端测试、供应商模拟桩和联调说明。

回滚策略：
- 若第三方数据集成阻塞，可先保留 Django + Agent 基础骨架，并隔离关闭特定适配器。
- 若前端执行过程 UI 效果不达标，可保留后端能力并逐步迭代 `./frotend` 中的可视化层。

## 开放问题

- 高德与百度地图所需的实际访问方式、凭证与额度应在实现前确认，并写入环境配置说明。
- 首版是否需要在前端提供来源筛选、刷新重试或仅展示自动选择后的结果。
- 旅行规划结果是否需要支持导出为 Markdown 或 PDF，还是首版只做页面展示。
