## 为什么

当前项目缺少一个能够把用户约束条件直接转化为可执行旅行方案的智能体能力。随着 LangChain 和 LangGraph 已成为构建多步骤 agent workflow 的常见方案，现在补齐这项能力，可以让后续实现围绕稳定的规划流程、状态管理和工具扩展逐步落地。

## 变更内容

- 在 `./backend` 下新增一个基于 Django、LangChain 与 LangGraph 的旅行规划 Agent 服务，用于接收预算、天数、目的地、出行偏好等输入并生成全面的旅行方案。
- 接入阿里云百炼千问模型 `qwen3.5-plus` 作为首版默认推理模型，并为模型配置、密钥管理和错误处理预留独立适配层。
- 接入高德地图 POI 搜索与百度地图地点检索等个人开发者可接入的数据源，为景点、美食、住宿候选提供尽可能真实、实时、可靠的数据支撑。
- 在 `./frotend` 下新增一个基于 Vue 3 的前端界面，用于收集旅行规划输入、展示 Agent 执行过程，并呈现最终旅行方案。
- 定义旅行规划流程的规范，包括输入收集、约束校验、外部数据检索、规划推理、结果校验、输出组装和失败兜底。
- 定义旅行方案输出的结构，覆盖行程摘要、逐日安排、预算拆解、交通与住宿建议、景点与美食推荐、来源信息、时间戳和注意事项。
- 定义蓝色系且具有专门交互设计的 UI 要求，包括表单、执行状态、数据源反馈、结果展示、加载态与错误态的视觉一致性。
- 为后续实现提供技术设计与任务拆分，确保可以逐步交付具备真实数据支撑的旅行规划系统。

## 功能 (Capabilities)

### 新增功能
- `trip-planning-agent`: 根据用户提供的预算、天数、目的地与偏好，结合实时检索结果生成结构化、可读的完整旅行方案。
- `travel-data-integrations`: 接入高德地图 POI 与百度地图地点检索等数据源，并将多源结果标准化为 Agent 可消费的数据；首版住宿采用地图酒店候选而非 OTA 实时库存。
- `trip-plan-output`: 定义旅行方案的标准输出结构，包括每日行程、预算分配、推荐建议、约束说明、数据来源和时间信息。
- `trip-planner-ui`: 定义 Vue 3 前端输入、Agent 执行过程与结果展示界面，包括蓝色系视觉风格、加载反馈、执行进度和异常提示。

## 影响

- 将在 `./backend` 中新增 Django 项目结构、旅行规划相关的领域模型、agent workflow、状态图、模型接入层、第三方数据适配层与 API。
- 将在 `./frotend` 中新增 Vue 3 页面、表单状态管理、执行过程可视化、结果渲染和蓝色系样式系统。
- 将引入 Python 生态的 LangChain/LangGraph、千问 `qwen3.5-plus`、高德地图与百度地图等集成依赖，并需要处理认证、限流、失败回退和数据标准化。
- 会影响前后端输入输出契约，因为旅行规划请求需要标准化输入结构，响应需要稳定的方案格式。
