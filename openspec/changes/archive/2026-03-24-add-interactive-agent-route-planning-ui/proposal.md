## 为什么

当前产品的主入口仍然是一次性表单提交，适合明确输入，但不适合真实旅行规划里常见的模糊需求、逐步澄清和路线协商。现在需要把系统扩展成以 Agent 交互式对话为主的路由规划工作台，让用户可以边聊边补充约束、收敛目的地与路线偏好，同时保留表单一键生成作为次级快捷入口。

## 变更内容

- 将前端主界面从“表单优先”调整为“Agent 对话优先”，默认展示可持续追问和澄清的交互式规划界面。
- 为旅行规划 Agent 增加多轮会话能力，使其能够基于用户逐步提供的预算、天数、目的地、交通方式、必去点位、同行人信息和节奏偏好，辅助生成更贴近真实需求的路由规划。
- 将现有“一键生成旅行方案”的表单入口保留为次级功能，作为快捷填报方式接入同一套规划后端。
- 扩展后端规划流程，使其能够处理对话上下文、澄清问题、阶段性建议和最终路线方案，而不只是一轮静态请求。
- 接入或扩展地图路径规划能力，用于生成点位之间的交通衔接建议、路线顺序和可替代路径，而不仅是通用旅行推荐。
- 丰富路线规划输出，突出路线顺序、点位衔接、区块切换、交通建议、时间段安排和可替代路径。
- 调整主界面信息架构和蓝色系 UI，使执行过程、会话历史、实时检索反馈和路线结果成为统一体验的一部分。

## 功能 (Capabilities)

### 新增功能
- `interactive-route-planning-chat`: 支持用户通过多轮对话逐步澄清出行目标、约束和点位偏好，由 Agent 辅助收集信息并生成结构化路由规划。
- `planner-primary-chat-ui`: 将对话式 Agent 工作台定义为主界面，整合会话区、执行过程区和路线结果区，并将表单一键生成降级为辅助入口。
- `route-directions-integration`: 结合地图路径规划能力生成点位之间的路线衔接、交通方式建议和替代路径信息。
- `route-planning-output`: 定义面向“路由规划”场景的结果结构，覆盖路线摘要、停靠点顺序、时段安排、交通衔接、替代方案和澄清记录。

## 影响

- 将影响 `./frotend` 的主页面结构、交互流程、视觉层次和状态管理，尤其是首页入口、聊天面板、执行时间线和结果展示布局。
- 将影响 `./backend` 中的请求 schema、LangGraph 工作流、规划上下文构建方式以及 API 契约，新增对多轮会话与澄清回合的支持。
- 将影响当前旅行规划输出格式，因为结果需要从“通用旅行方案”扩展为“带路线顺序、路径衔接与替代路线的规划结果”。
- 将影响地图集成层，因为点位检索之外还需要路径规划或路段时间估算能力来支撑路线建议。
- 将影响前后端联调方式和测试范围，需要覆盖对话输入、澄清回合、会话状态恢复、主次入口切换和路线结果展示。
