## 上下文

当前系统已经具备一次性表单提交、实时数据检索、LangGraph 规划流程和结果展示能力，但其产品心智仍然是“提交表单后等待一次结果”。这对需求明确的用户有效，但对真实旅行规划中常见的模糊起点并不友好，例如“想在上海逛一天，边走边吃，不想太赶”“我想从博物馆一路逛到外滩，中间穿插咖啡馆”等场景。

现状限制主要有三点：
- 后端请求模型是单轮静态输入，缺少会话、澄清和多轮修订能力。
- 前端首页以表单为主，无法承载连续对话、上下文确认和增量式路线调整。
- 路径建议主要来自模型组织已有 POI 候选，缺少对“点位之间如何走”的路线级数据支撑。

这次变更不是替换现有规划能力，而是在现有旅行规划骨架上增加一层“对话式路由规划”体验，使系统从一次性生成器升级为可协商、可修订、可追问的 Agent 工作台。

涉及的主要利益相关者：
- 终端用户：需要更自然的交互方式来逐步收敛路线。
- 前端实现者：需要把聊天、执行状态、结果面板整合成一个主工作台。
- 后端实现者：需要在现有 LangGraph 和 API 之上增加会话状态、澄清回合和路径规划整合。

## 目标 / 非目标

**目标：**
- 提供一个以对话式 Agent 为主入口的路线规划工作台，用户可通过多轮对话逐步补齐需求。
- 让 Agent 能够记住已确认的预算、目的地、点位、交通偏好和节奏要求，并据此迭代路线方案。
- 在现有检索能力之上补充路径规划或路段估算能力，使结果具备“从 A 到 B 如何走”的路线逻辑。
- 保留现有表单入口，但将其定位为辅助入口，并复用同一规划核心。
- 扩展输出结构，使结果既能表达最终路线，也能表达澄清问题、路线调整和候选替代路径。

**非目标：**
- 不实现用户登录、长期会话持久化、多设备同步或历史会话云端归档。
- 不在首版支持语音输入、语音播报或富媒体消息。
- 不实现真正的导航级逐转指令，而是提供路线级衔接建议和交通方案。
- 不在首版引入新的复杂前端状态库；优先沿用当前 Vue 组合式状态管理。
- 不移除现有表单规划 API；其仍保留为兼容和快捷入口。

## 决策

### 决策 1：新增“会话 + 回合”模型，而不是把聊天内容硬塞进现有单轮请求

后端将新增对话会话实体，用于记录：
- `session_id`
- 消息列表（用户消息、助手消息、系统澄清）
- 已确认约束快照
- 最新路线草案或最终路线
- 当前状态（澄清中 / 规划中 / 已完成 / 失败）

理由：
- 当前 `TripPlanningRequest` 适合单次提交，不适合承载多轮上下文。
- 把聊天文本直接拼成 `notes` 会让澄清逻辑、修订逻辑和结果回写不可测试。
- 显式会话模型有利于前端展示聊天记录和当前已确认条件。

备选方案：
- 继续只用单轮请求，每次让前端传递完整上下文字符串。实现快，但消息边界和已确认约束无法结构化维护。
- 直接引入数据库持久化会话。长期更稳，但超出本次主目标，首版可先沿用内存存储。

### 决策 2：保留现有 run/polling 模式，但将其扩展到聊天回合

聊天回合仍沿用当前“创建任务后轮询结果”的执行模式，而不是这次同时切换到 WebSocket 或 SSE。

做法：
- 新增聊天回合 API，用于提交一条用户消息并创建一个新的规划回合。
- 每个回合同样产出执行步骤、结果状态和结构化返回。
- 前端聊天面板在发送后轮询该回合状态，并在完成后把助手消息与路线结果更新到界面。

理由：
- 现有后端已经围绕 `PlannerRunStore` 和 polling 运行，改造成本最低。
- 本次核心问题是交互模型和路线规划，不是传输协议升级。
- 先复用 polling 可以缩小变更面，并保证表单入口与聊天入口共享同一套运行机制。

备选方案：
- 直接切换流式响应。体验更好，但会显著增加前后端复杂度和调试面。

### 决策 3：聊天入口和表单入口共享同一条“规划核心”，只在输入收集方式上分叉

系统将抽象出统一的规划上下文模型，聊天入口与表单入口都先转换为同一结构，再进入规划流程。

规划核心负责：
- 标准化约束
- 检索 POI / 餐饮 / 酒店 / 路径数据
- 生成路线方案
- 校验预算、时间、路径顺序和可解释性

聊天入口额外负责：
- 识别缺失信息
- 生成澄清问题
- 合并上一轮确认的约束
- 接受用户对既有草案的修订请求

理由：
- 如果聊天和表单各走一套规划逻辑，后续维护会迅速分叉。
- 主入口改变的是交互层，不应复制底层规划能力。

备选方案：
- 完全独立的 chat workflow。灵活，但会和现有单轮规划能力脱节。

### 决策 4：引入地图路径规划能力生成路段级建议

系统将在现有地图 POI 检索基础上，增加点位之间的路径衔接能力。首选沿用已接入的地图供应商路径规划接口，以减少额外供应商耦合。

路径规划层输出至少包含：
- 起点与终点
- 推荐交通方式
- 估算时长
- 备注或切换建议

当路径接口不可用时：
- 使用区域距离、地铁优先策略和点位聚类结果给出启发式衔接建议
- 明确标注为降级路线估算

理由：
- “路由规划”与“通用行程推荐”的关键区别就在于点位之间的衔接逻辑。
- 只给出点位列表而没有路段说明，会让对话式规划缺乏实际执行价值。

备选方案：
- 完全依赖模型根据 POI 文本自行编排路段。实现快，但可靠性差。
- 直接接入更专业的导航服务。能力更强，但超出首版范围。

### 决策 5：前端首页改为聊天工作台，表单以抽屉或折叠区作为次级入口

首页信息架构将调整为：
- 主区：聊天会话区，承载用户消息、助手澄清与路线答复
- 辅区：执行过程区，展示当前回合检索与规划步骤
- 结果区：路线卡片、路段衔接、预算与来源信息
- 次级入口：一键生成表单，以抽屉、折叠面板或标签页形式出现

理由：
- 如果聊天只是表单旁边的小组件，就无法形成“主界面”。
- 路线规划需要会话上下文、执行过程和结果区联动展示，适合工作台而不是单卡片页。

备选方案：
- 首页继续表单优先，聊天作为增强入口。实现快，但与本次目标相悖。

### 决策 6：结果结构扩展为“澄清态 + 草案态 + 最终态”三层

输出需要支持三类状态：
- 澄清态：还缺关键信息，只返回澄清问题和已确认约束
- 草案态：已有可读路线，但允许用户继续修订
- 最终态：已确认路线，输出完整路段、点位和交通建议

新增或强化字段建议包括：
- `conversation_summary`
- `confirmed_constraints`
- `route_overview`
- `route_stops`
- `route_legs`
- `alternatives`
- `clarification_questions`
- `revision_notes`

理由：
- 聊天式规划并不是每轮都直接产出完整终稿。
- 没有中间态定义，前端就只能把每轮都当作最终结果，体验会失真。

备选方案：
- 始终只返回最终态格式。简单，但不适合多轮协商。

## 风险 / 权衡

- [会话状态仅保存在内存中，服务重启后会丢失] → 在本次设计中明确这是首版权衡，并保证表单入口与最终结果不依赖长期会话。
- [路径规划接口可能限流或不可用] → 对单个路段做降级估算，并在结果中明确标注来源与不确定性。
- [聊天主界面会让前端状态复杂度上升] → 约束为单会话、单活动回合，不在首版支持并行会话窗口。
- [多轮交互可能拉长用户等待时间] → 保留执行时间线与阶段反馈，并允许在部分信息已足够时输出路线草案。
- [聊天输入更自由，模型更容易误解] → 增加“已确认约束”摘要与澄清回合，避免直接根据模糊文本生成终稿。

## 迁移计划

1. 新增聊天会话相关 schema、API 和内存存储实现。
2. 抽象统一规划上下文，把表单提交和聊天回合都映射到同一规划核心。
3. 为地图集成层增加路径规划适配器或路段估算服务。
4. 扩展 LangGraph 节点，支持澄清判断、上下文合并、路线草案生成和修订。
5. 重构首页为聊天主工作台，并把表单入口下沉为次级面板。
6. 更新结果展示结构、测试与联调说明。

回滚策略：
- 若聊天主界面未完成，可暂时隐藏新入口并保留表单优先页面。
- 若路径规划适配器不稳定，可回退到启发式路段估算，同时保留会话能力。
- 若聊天状态管理影响稳定性，可先只开放“单轮聊天转结构化表单”的简化模式。

## 开放问题

- 首版聊天会话是否需要跨页面刷新保留，还是仅在单页生命周期内有效。
- 路径规划优先使用哪种模式：步行、公共交通、驾车，还是根据用户偏好动态选择。
- 是否要在前端允许用户手动调整站点顺序并触发重新规划，还是先只支持通过自然语言修订。
