## 上下文

当前系统已经拥有较完整的产品基础：对话式旅行规划主工作台、流式执行反馈、路线结果展示、游客与登录态切换、会话与方案资产化，以及后端提供的身份摘要和能力开关。问题不在于能力缺失，而在于这些能力分散在多个页面和多个表达层里，导致用户容易把产品理解成“带账号体系的旅行工具集合”，而不是“帮助我持续收敛并管理旅行决策的 Agent 工作台”。

这轮变更主要涉及前端产品信息架构，但不能只停留在文案层，因为现有后端已经通过 `auth/me`、会话列表、方案列表和流式/详情接口暴露了足够多的连续性信息。如果前端仍然按页面各自取数、各自展示，产品心智会继续割裂。利益相关者包括：首次进入产品的用户、已沉淀历史资产的回访用户、前端工作台状态管理、以及需要稳定输出身份摘要和连续性字段的后端接口。

## 目标 / 非目标

**目标：**
- 统一产品级表达，让“个人旅行规划 Agent + 资产化工作台”成为首页、工作台、资产页和账户入口的一致心智。
- 让主工作台从“聊天区 + 结果区 + 执行区”的平级展示，收敛为围绕“当前阶段”和“下一步动作”的决策流。
- 强化会话页、方案页和个人中心的连续性表达，让用户能迅速判断从哪里继续、为什么停在这里、哪一版最接近最终出行版本。
- 将后端已有的 `asset_summary`、`capabilities`、最近会话等摘要能力接入前端关键页面，避免重复拉取和重复推断。
- 调整导航层级与页面角色，减少后台式结构，让资产与账户能力服务于主任务，而不是与主任务竞争注意力。

**非目标：**
- 不在本次变更中引入 OTA 预订、下单支付、日历同步或多人协作。
- 不重构 LangGraph 规划核心、地图检索能力或流式协议本身。
- 不新增复杂的分享社区、模板市场或公开方案广场。
- 不把整个产品改造成地图优先或表单优先体验，聊天仍然是主入口。

## 决策

### 1. 将产品统一表述为“持续打磨旅行决策的 Agent 工作台”
- 决策：产品文案、导航标签、空状态和登录引导必须统一强调“先对话收敛，再沉淀为资产，再持续修订”的主线，而不是泛泛强调“聊天”“旅游方案”或“账号功能”。
- 原因：当前已有会话和方案资产化能力，但若缺少统一叙事，用户会把这些理解为附属后台而非主价值的一部分。
- 备选方案：
  - 继续保留当前各页独立文案：实现成本低，但会继续稀释产品心智。
  - 直接转向“旅行资产管理器”定位：会弱化 Agent 规划的首要价值。

### 2. 工作台采用“阶段摘要 + 下一步动作 + 支撑面板”的结构
- 决策：工作台保留聊天、结果和执行过程三类信息，但视觉和交互层必须围绕统一的阶段摘要与下一步动作收拢，而不是默认让三块模块平权竞争注意力。
- 原因：用户最关心的不是“页面里有哪些区块”，而是“系统现在知道什么、缺什么、推荐我下一步做什么”。
- 备选方案：
  - 继续优化现有三栏/双栏布局：能改善视觉，但无法从根本上解决决策感不足的问题。
  - 全部压成单线程聊天流：会牺牲路线结果与执行轨迹的可消费性。

### 3. 资产页必须从 CRUD 列表升级为“连续性入口”
- 决策：会话页、方案页和个人中心都必须显示停留原因、最新状态、恢复入口、版本语义与推荐下一步，而不是只展示标题、状态和删除/继续按钮。
- 原因：资产页的任务不是“存放记录”，而是帮助用户低成本恢复上下文，继续做旅行决策。
- 备选方案：
  - 维持列表 + 操作按钮：实现简单，但只能服务熟悉系统的用户。
  - 把所有连续性信息都放回工作台：会让历史资产页变成弱价值页面。

### 4. 前端优先消费后端已有身份摘要接口，而不是继续分散拉取与本地推断
- 决策：以 `GET /api/auth/me/` 返回的 `asset_summary`、`capabilities` 和 guest 状态作为产品级状态源，驱动顶部提示、个人中心摘要、游客迁移引导和部分工作台动作可见性。
- 原因：后端已经把身份摘要聚合好了，前端继续各页单独请求列表再自行推断，会制造不一致与额外加载成本。
- 备选方案：
  - 保持现有页面分别拉取 sessions/plans：短期可行，但产品摘要层永远滞后于真实资产状态。

### 5. 收敛导航，把账户管理下沉为支撑能力
- 决策：一级导航必须围绕主工作流组织，账户相关能力应尽量靠近用户菜单或资产摘要，而不是作为与工作台并列的重入口。
- 原因：对当前阶段产品而言，账户不是主任务，旅行规划与继续修订才是主任务。
- 备选方案：
  - 保持四个一级入口：信息完整，但主心智不聚焦。
  - 完全取消单独的账户页：会让密码设置、迁移说明和身份摘要缺少稳定承载面。

## 风险 / 权衡

- [工作台结构调整后，已有用户需要重新适应] → 保持聊天主入口和结果/执行面板能力不变，只重排优先级与文案表达。
- [前端开始依赖 `auth/me` 聚合摘要，可能引入更多初始化耦合] → 将摘要作为增强层，列表接口仍保留为详细数据源和回退路径。
- [资产页信息密度提高，可能增加实现复杂度] → 优先展示“停在哪里、接下来做什么、哪条最值得继续”三类高价值信息，避免一次性堆砌所有元数据。
- [导航收缩会引发页面归属调整] → 用明确的信息架构规则区分“主任务入口”和“支撑配置入口”，避免反复摇摆。
- [这轮更偏产品与信息架构，容易被误解为纯视觉优化] → 在 specs 和 tasks 中明确要求状态摘要、连续性提示和能力消费等可测试行为，而不是只写样式任务。

## Migration Plan

1. 先定义产品定位、工作台决策流、资产连续性和身份摘要消费的行为规范，确保这轮优化不退化成纯文案重写。
2. 在前端壳层与关键页面引入统一的产品表达和摘要消费逻辑，同时保留现有 sessions/plans 详细接口作为内容源。
3. 分阶段重构工作台、会话页、方案页和个人中心，优先保证“继续规划”主链路不受影响。
4. 用组件/页面测试覆盖游客态、登录态、最近会话恢复、下一步动作提示和继续修订路径，确保心智优化不带来行为回退。
5. 若某些摘要字段不足以支撑前端展示，再最小化补充后端接口字段，避免破坏现有契约。

## Open Questions

- 个人中心是否应保留为一级页面，还是转为以用户菜单打开的二级界面。
- “下一步动作”在工作台中应表现为独立固定区域，还是嵌入结果区顶部摘要中。
- 会话页与方案页是否需要“推荐继续项”排序策略，例如优先显示等待澄清、最近生成、接近最终版的条目。
- 后端是否需要为方案列表补充“最近修改原因”或“来源会话摘要”等字段，以更好支持连续性表达。
