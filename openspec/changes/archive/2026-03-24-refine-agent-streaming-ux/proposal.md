## 为什么

当前产品虽然已经具备对话式旅游规划、结果面板和执行过程展示，但体验层仍然存在明显割裂：聊天区上方信息噪声较多、关键按钮过于突兀、结果模块样式有缺陷且层级不稳定，执行流程与结果之间也缺少清晰的主次关系。同时，对话接口仍以轮询式完成态返回为主，无法支撑真正自然的流式交互反馈。现在需要做一轮阶段性重构，把“旅游规划 Agent”定位、流式对话体验、结果可读性和接口稳定性整体收敛成一套更统一的产品体验。

## 变更内容

- 重构前端工作台信息架构，弱化聊天窗口上方的冗余标题信息，并把顶部操作按钮改为更克制、更不打扰主任务流的展示方式。
- 全面审查并修复结果区现有样式和布局问题，统一模块间距、状态样式、信息层级和响应式行为，避免结果卡片、摘要、路线、预算、替代方案等区域出现视觉断裂或错位。
- 重构结果区与执行过程区的展示方式，让用户能够更直观地区分“当前执行到哪一步”“已经产出了哪些可消费结果”“哪些内容仍在生成/澄清”。
- 引入对话级流式返回能力，使聊天回复、执行状态和阶段性结果能够渐进展示，而不是等整轮任务结束后一次性刷新。
- 对现有后端接口和已实现的技术场景进行阶段性审查，补齐不清晰的返回契约、状态表达、错误语义和流式协同路径，为前端统一展示提供稳定接口基础。
- 在产品表达上进一步凸显“旅游规划 Agent”定位，把用户在不同阶段最关心的信息放在最显著的位置，并收敛不同阶段下界面的视觉语言与交互模式。

## 功能 (Capabilities)

### 新增功能
- `streaming-chat-delivery`: 提供面向聊天会话的流式消息与阶段状态返回能力，使回复、步骤进展和阶段性结果可以渐进到达前端。
- `agent-workbench-experience`: 定义聚焦旅游规划 Agent 的主工作台体验，弱化冗余头部信息，收敛顶层操作按钮和多阶段界面结构。
- `result-and-workflow-visibility`: 定义结果区与执行过程区的统一展示规范，确保用户始终清晰理解当前进展、阶段结果和下一步动作。
- `backend-interface-review`: 规定对现有后端接口、场景处理和错误/状态契约的系统性审查与优化要求。

## 影响

- 将影响 `./frotend/src` 下的工作台页面、聊天组件、结果面板、执行时间线、样式系统以及相关状态管理和测试。
- 将影响 `./backend/planner/api`、`./backend/planner/services`、规划执行链路以及与前端联调的返回契约，尤其是流式传输、阶段状态表达和错误语义。
- 将影响现有用户交互路径，因为聊天、执行过程和结果消费将被重组为更加统一的“阶段感知式”体验。
- 将影响接口验证、前端展示测试和端到端联调范围，需要覆盖流式回复、执行过程渐进更新、结果模块稳定展示与不同阶段 UI 收敛。
