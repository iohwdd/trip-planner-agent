## 新增需求

### 需求:聊天回合必须支持流式事件返回
系统必须为聊天会话提供流式事件返回能力，使前端在单个回合执行期间可以逐步接收回复内容、阶段状态和完成信号，而不是只能等待整轮任务结束后再获取结果。

#### 场景: 发起流式聊天回合
- **当** 客户端以流式模式提交一个聊天消息或表单回合
- **那么** 系统必须立即建立可持续读取的响应通道并开始输出该回合的事件流

#### 场景: 流式返回阶段状态
- **当** 回合进入排队、执行、澄清、完成或失败阶段
- **那么** 系统必须输出带有统一阶段状态字段的事件，供前端直接驱动阶段 UI

### 需求:流式事件必须包含可区分的事件类型
系统必须为流式输出定义稳定的事件类型，至少覆盖消息增量、步骤状态、阶段结果、回合完成和回合错误，前端禁止依赖非结构化字符串猜测事件含义。

#### 场景: 推送消息增量
- **当** 助手回复内容以渐进方式产生
- **那么** 系统必须发送可追加到当前助手消息的增量事件，并附带回合标识和顺序信息

#### 场景: 推送工作流步骤更新
- **当** 工作流中的某个步骤开始、降级、完成或失败
- **那么** 系统必须发送结构化步骤事件，至少包含步骤 key、标题、状态和可选说明

### 需求:流式与非流式接口必须保持状态语义一致
系统必须保证新增流式接口与现有详情/轮询接口在回合状态、错误语义和结果完成态上保持一致，避免同一回合在不同消费方式下出现语义冲突。

#### 场景: 流式完成后拉取详情
- **当** 客户端在流式回合完成后再次查询会话或回合详情
- **那么** 返回的状态和结果必须与流式通道最后一个完成事件保持一致
