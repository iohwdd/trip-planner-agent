## 上下文

当前前端工作台已经具备聊天、执行过程和结果展示三块能力，但它们在布局和状态层面仍然相互割裂：聊天顶部信息和操作区挤占主要注意力，结果区模块视觉不一致，执行过程与最终结果之间缺少统一的阶段语义。后端虽然已支持会话、回合、任务和结果持久化，但聊天回复仍以“提交后轮询完成态”的模型为主，无法把模型回答、步骤进展、澄清状态和阶段性结果流式送达前端。此次重构横跨前端信息架构、设计系统、接口契约和聊天传输模式，属于典型的跨模块设计变更。

利益相关者主要包括：最终用户（需要更清晰、更可信的旅行规划体验）、前端工作台（需要稳定且渐进的展示数据）、后端规划服务（需要表达中间状态与流式事件）、产品定位（需要突出旅游规划 Agent 主价值，而不是通用聊天框）。

## 目标 / 非目标

**目标：**
- 建立统一的“阶段感知式”工作台结构，让用户在聊天、执行、结果三个阶段都能迅速看到最重要的信息。
- 将聊天回复升级为流式传输，使消息文本、工作流步骤和阶段性结果可以增量刷新。
- 审查并收紧现有后端接口的状态、错误和结果契约，减少前端需要猜测或二次推导的情况。
- 修复结果模块在不同数据状态、不同屏幕尺寸下的样式和层级问题，统一视觉和交互模式。
- 凸显“旅游规划 Agent”定位，把用户关心的约束确认、路线摘要、执行进度和下一步动作放在主要视区。

**非目标：**
- 不在本次变更中引入新的规划模型供应商或替换现有 LangGraph 工作流核心。
- 不重做账号体系、方案管理或数据库结构的大规模迁移。
- 不在本次变更中实现完整的多人协作、实时共享或 WebSocket 基础设施扩展到所有接口。
- 不追求一次性统一所有历史页面，仅聚焦主工作台、相关结果展示和聊天执行链路。

## 决策

### 1. 采用基于 HTTP 流的会话事件协议，而不是继续仅靠轮询
- 决策：为聊天回合新增流式响应路径，优先采用浏览器友好的 HTTP streaming / SSE 风格事件输出，事件类型覆盖消息增量、步骤状态、阶段结果、回合完成和错误。
- 原因：相比“提交 -> 轮询 -> 整体替换”，事件流更贴近用户预期，也更容易让前端在一个统一通道中展示文本进展、执行轨迹和阶段结果。
- 备选方案：
  - 继续轮询：实现简单，但无法自然表达 token 级消息和细粒度过程更新。
  - 直接上 WebSocket：双向能力更强，但当前需求主要是服务端到客户端的单向推送，引入复杂度较高。

### 2. 将工作台重构为“主内容 + 阶段上下文”而不是多个平级区域竞争注意力
- 决策：聊天区保持主入口地位，但移除或弱化聊天窗口上方的大标题信息；顶部按钮改为低干扰、轻量化操作；结果和执行过程收敛为统一的上下文面板/阶段视图，而不是多个视觉重量相近的模块平铺竞争。
- 原因：用户最关心的是当前对话、现在系统在做什么、已经产出什么，而不是重复性的页头标签。
- 备选方案：
  - 继续保留当前多区块平铺：实现成本低，但层级持续混乱。
  - 全屏聊天 + 抽屉结果：会弱化结果消费和过程可见性，不适合旅游规划场景。

### 3. 结果展示采用“摘要优先 + 分层展开”而不是所有卡片同权重堆叠
- 决策：结果区优先展示路线摘要、当前状态、关键约束、下一步建议，再按主题拆分路线停靠点、交通衔接、预算、替代方案、修订说明和数据来源等模块，并统一空态、加载态、澄清态与降级态样式。
- 原因：旅游规划结果并不是一组等权信息，摘要、阶段状态和路线主线必须优先。
- 备选方案：
  - 保持当前模块顺序不变：无法解决重点不突出问题。
  - 只显示摘要，细节另开页面：会增加跳转成本，削弱连贯体验。

### 4. 后端接口统一表达阶段状态、消费提示和错误语义
- 决策：现有轮询/详情接口与新增流式事件都使用统一的阶段状态词汇，明确区分 queued、running、clarification、partial、completed、failed 等阶段；错误返回保留用户可理解错误文案和结构化 code/source 字段；结果快照标明是否为阶段性输出。
- 原因：前端在不同接口上必须得到同构状态，才能稳定驱动统一 UI。
- 备选方案：
  - 前端自行映射现有各类状态：短期可行，但技术债会持续增加。

### 5. 用一次审查顺带收敛已有技术场景的实现边界
- 决策：在改造过程中同步审查会话恢复、澄清回合、失败降级、结果保存前提、流式中断恢复等场景，并在 design/tasks 中明确测试覆盖。
- 原因：流式化会放大原有状态边界问题，如果不同时审查，前后端体验会出现更明显的不一致。

## 风险 / 权衡

- [流式接口增加实现复杂度] → 先采用单向事件流，限制事件类型和节奏，避免首版过度泛化。
- [前端同时维护流式与非流式路径] → 让流式成为聊天主路径，保留现有轮询接口作为回退和测试基线。
- [结果区重构可能引入样式回归] → 增加页面级和组件级测试，重点覆盖空态、澄清态、执行中、完成态和移动端布局。
- [后端阶段状态扩展可能影响旧前端调用] → 保持关键已有字段兼容，只新增更细的状态和摘要字段，不做破坏式删除。
- [模型或外部数据源无法稳定提供细粒度进度] → 把流式拆分为“工作流节点级 + 文本增量级”，即便模型文本无法精细分片，也能先稳定推送步骤与阶段状态。

## Migration Plan

1. 先定义 proposal/specs 中的目标能力和接口要求，固定事件模型、状态词汇与前端阶段展示约定。
2. 在后端为聊天回合增加流式输出端点或可协商的流式模式，同时保持现有详情查询接口可用。
3. 在前端接入流式通道，重构聊天区、上下文区和结果区的状态组织方式，并修复现有样式缺陷。
4. 通过组件测试、页面测试和后端 API/工作流测试验证澄清、执行中、完成、失败、降级等典型路径。
5. 保留回退路径：若流式不可用，前端仍可退回现有轮询查询，以降低发布风险。

## Open Questions

- 首版流式接口是否采用单独的新端点，还是在现有提交接口上通过请求头/查询参数协商流式模式。
- 文本增量是否需要 token 级别粒度，还是先以句段/阶段性块为单位即可满足体验目标。
- 执行过程是否要在流式里发送完整结果快照，还是发送增量摘要并让前端在完成后拉取详情。
- 是否需要为移动端定义与桌面端不同的上下文面板默认展开策略。
