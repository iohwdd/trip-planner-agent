## 上下文

当前仓库已经具备以下基础：
- Django 单体后端，统一管理认证、会话、助手问答和工作台规划能力。
- 自定义 `User` 模型，已包含 `is_staff` 和 `is_superuser` 字段，可直接承载管理员权限。
- 智能助手已经拥有独立的会话/消息模型与 SSE 流式输出链路。
- 现有产品壳层采用统一身份摘要驱动导航与关键入口可见性。

这次变更同时跨越前端导航、管理员页面、对象存储、向量存储、RAG 检索编排与助手生成链路，属于明显的跨模块设计变更。已知约束如下：
- 知识库只服务于智能助手，不接入工作台 LangGraph 流程。
- 仅支持 `md` 和 `txt` 两种文档类型。
- 原始文件必须保存在 MinIO，对象元数据落库，向量信息持久化到 ChromaDB。
- 前端不需要检索调试界面，也不展示来源引用。

## 目标 / 非目标

**目标：**
- 提供一个仅 `is_staff` 用户可见和可访问的知识库模块，作为主导航一级入口。
- 允许管理员上传、查看、删除并重建 `md` / `txt` 文档索引。
- 为智能助手引入可持久化的 RAG 检索增强能力，并保持现有流式回复接口不变。
- 在统一身份摘要中暴露管理员能力开关，供前端决定导航与页面可见性。
- 将原始文件、文档元数据和向量索引分层保存，便于后续运维和重建。

**非目标：**
- 不为工作台规划流程注入知识库检索节点。
- 不支持 `pdf`、`docx`、网页抓取、OCR 或联网知识源。
- 不提供多知识库切换、租户隔离或普通用户自建知识库能力。
- 不在前端展示检索来源、引用卡片或检索调试视图。

## 决策

### 决策 1：首版采用“单默认知识库”模型
- 方案：后端维护一个默认知识库记录，管理员管理的所有文档都挂在该知识库下；前端页面也只围绕这个默认知识库展开。
- 原因：当前产品阶段并不需要多知识库切换，单库模型能降低前后端复杂度、权限判断和 Chroma 集合管理成本。
- 替代方案：
  - 多知识库模型：更灵活，但会引入多集合切换、默认库选择、跨库检索和更重的管理 UI，不适合首版。

### 决策 2：原始文件保存到 MinIO，文档元数据落 MySQL，向量索引保存到 ChromaDB
- 方案：
  - MinIO 保存原始 `md` / `txt` 文件。
  - Django 模型保存知识库和知识文档元数据，包括对象键、状态、错误信息和 chunk 数。
  - ChromaDB PersistentClient 保存 chunk 与 embedding。
- 原因：将“原件、管理元数据、检索索引”分层后，重建索引和删除文档都更可控，也符合对象存储和向量数据库的职责边界。
- 替代方案：
  - 本地磁盘保存原件：部署耦合更高，不利于迁移和备份。
  - MySQL 冗余保存全部 chunk 正文：结构冗余，删除/重建更复杂。

### 决策 3：文档入库采用内联处理，但保留显式状态字段
- 方案：管理员上传文档后，请求内直接完成 MinIO 上传、内容解析、切分、embedding、写入 Chroma 和状态更新；模型中仍保留 `pending / processing / ready / failed` 状态。
- 原因：当前仓库虽然已有规划任务队列，但知识库只面向管理员低频操作，且仅支持 `md` / `txt`，首版以内联处理更容易落地和验证。显式状态字段为后续切换到异步任务保留演进空间。
- 替代方案：
  - 完整异步 ingestion worker：更适合大规模文档，但会显著增加 worker 编排、轮询状态和失败恢复复杂度。

### 决策 4：Markdown 和文本采用不同的切分策略
- 方案：
  - `txt`：按中文友好的递归切分器处理，优先使用段落、换行和中文标点。
  - `md`：先按 Markdown 标题层级切段，再对子段递归切分，并在 metadata 中记录 `heading_path`。
  - 初始参数：`chunk_size=600`、`chunk_overlap=100`。
- 原因：Markdown 文档保留章节上下文有助于检索质量；纯文本则不需要额外结构解析。
- 替代方案：
  - 统一纯文本切分：实现更简单，但会损失 Markdown 章节语义。

### 决策 5：助手 RAG 链路采用“先召回再重排再生成”的最小闭环
- 方案：
  - 使用 `shibing624/text2vec-base-chinese` 生成 embedding。
  - Chroma 先召回 `top_k=12`。
  - 用 `cross-encoder/mmarco-mMiniLMv2-L12-H384-v1` 重排并保留 `top_n=4`。
  - 将重排后的 chunk 拼入现有助手 prompt 上下文，再走现有 Qwen 流式生成链路。
  - 若知识库为空或检索结果不足，则回退到普通助手回答。
- 原因：这条链路是当前需求的最小可用 RAG 闭环，能够在不破坏现有 SSE 协议的前提下增强回答质量。
- 替代方案：
  - 只做向量召回不重排：实现更轻，但相关性稳定性更差。
  - 直接替换生成模型或独立 RAG 服务：成本更高，也不符合当前单体架构。

### 决策 6：管理员能力通过统一身份摘要向前端暴露
- 方案：在 `auth/me` 和现有 identity summary 中新增 `can_manage_knowledge_base` 能力位，值直接映射 `actor.user.is_staff`。
- 原因：当前壳层导航和关键页面已经依赖统一身份摘要驱动，新增管理员入口也应遵循同一机制，避免前端分散判断用户字段。
- 替代方案：
  - 前端直接读取 `user.is_staff`：耦合认证细节，破坏现有能力开关模式。

## 风险 / 权衡

- [风险] 首次加载 embedding 和 reranker 模型耗时较长，上传接口响应可能变慢
  → 缓解措施：采用惰性单例缓存模型实例，并将首版文档类型收敛到 `md` / `txt`，后续如需要可平滑切换到异步任务。

- [风险] `cross-encoder/mmarco-mMiniLMv2-L12-H384-v1` 是多语言模型，中文排序效果可能不如专门中文 reranker
  → 缓解措施：首版按指定模型落地，并通过独立 reranker 服务封装保留后续替换空间。

- [风险] MinIO 上传成功但 Chroma 写入失败，可能产生“有原件、无索引”的中间态
  → 缓解措施：文档模型记录失败状态和错误信息；管理员可手动重建索引；删除文档时同时清理对象与向量。

- [风险] 当前前端不展示来源引用，用户无法直接感知答案是否来自知识库
  → 缓解措施：后端保留检索命中 metadata 到消息层，后续如需审计或展示来源，不必重构主链路。

- [风险] Chroma 本地持久化目录需要部署侧保留卷
  → 缓解措施：通过环境变量配置持久化目录，并在设计中明确这属于知识库运行前置条件。

## Migration Plan

1. 增加知识库相关 Django 模型和迁移，不改动既有助手与工作台数据。
2. 引入 MinIO、Chroma、embedding 与 reranker 相关依赖和运行配置。
3. 新增知识库服务层、API 与管理员权限校验。
4. 新增前端知识库页面与壳层导航可见性逻辑。
5. 将助手消息生成链路改为“优先尝试知识库检索增强，失败则回退普通回答”。
6. 验证管理员上传、删除、重建索引和普通用户助手问答路径。

回滚策略：
- 若知识库链路异常，可临时关闭知识库配置或在服务层直接回退普通助手回答。
- 若前端管理员页存在问题，可移除导航入口而不影响普通用户使用智能助手。
