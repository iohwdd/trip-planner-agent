## 新增需求

### 需求:助手域与规划域必须在 API 命名空间上分离
系统必须为智能助手和流程化规划提供独立的 API 命名空间，禁止继续让一个 `/chat` 域同时承担助手和规划职责。

#### 场景:前端请求助手能力
- **当** 前端访问智能助手相关接口
- **那么** 请求必须落到独立的助手 API，而不是旅行规划 API

#### 场景:前端请求工作台规划能力
- **当** 前端提交结构化规划请求
- **那么** 请求必须落到规划运行 API，并返回规划运行状态与结果

### 需求:旅行方案资产必须支持来自工作台运行的来源追踪
系统必须记录旅行方案的来源是聊天规划还是工作台运行，以支持前端正确表达来源上下文。

#### 场景:从工作台保存方案
- **当** 用户将一次结构化规划运行保存为方案
- **那么** 系统必须记录该方案来自对应的规划运行

### 需求:新前端主链路不得依赖旧聊天规划接口
系统必须保证新前端主链路完全建立在助手域 API 和规划域 API 之上，旧聊天规划接口仅可作为兼容能力保留。

#### 场景:用户使用新版本前端
- **当** 用户通过导航进入智能助手或流程工作台
- **那么** 前端不得再调用旧的聊天规划主链路接口完成核心流程
