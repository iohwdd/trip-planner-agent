# trip-session-management 规范

## 需求
### 需求:登录用户的旅行会话和回合必须持久化保存
系统必须把登录用户的旅行会话、消息历史、已确认约束和每个回合的结果保存为可追溯资产，而不是仅保存在进程内内存中。

#### 场景:用户完成一次对话规划后刷新页面
- **当** 登录用户完成一个包含多轮消息的旅行会话并刷新页面
- **那么** 系统必须能够重新加载该会话的消息历史、已确认约束和最新结果
- **那么** 系统不得因为页面刷新而丢失历史回合

### 需求:系统必须提供会话列表并支持继续编辑
系统必须允许登录用户查看自己的会话列表，并从任意历史会话继续发起新的对话修订。

#### 场景:用户从历史会话继续编辑
- **当** 登录用户在“我的会话”中打开一个旧会话
- **那么** 系统必须展示该会话的消息历史和最近一次路线结果
- **那么** 用户发送新消息后必须在同一会话内生成新的回合

### 需求:聊天输入与快捷表单必须进入同一会话
系统必须将聊天输入和快捷表单提交统一写入当前活动会话，而不得把表单生成结果保存为另一套独立记录。

#### 场景:用户在聊天后使用快捷表单补充预算
- **当** 用户在当前会话中打开快捷表单并提交天数、预算和兴趣等条件
- **那么** 系统必须在当前会话中新增一个表单回合
- **那么** 前端结果区和消息区必须继续围绕同一会话更新

### 需求:用户必须能够重命名和删除自己的会话
系统必须允许登录用户管理自己的会话标题，并删除不再需要的历史会话。

#### 场景:用户整理自己的会话列表
- **当** 登录用户修改某个会话标题或删除一个旧会话
- **那么** 系统必须只影响该用户所选中的会话
- **那么** 列表页和工作台中的会话状态必须同步更新

### 需求:系统必须恢复最近会话
系统必须支持最近会话恢复：登录用户至少基于服务端历史恢复最近会话，游客至少基于浏览器本地状态恢复最近会话。

#### 场景:登录用户重新进入工作台
- **当** 登录用户重新打开工作台首页
- **那么** 系统必须能够自动恢复最近访问的会话或明确展示最近会话入口
- **那么** 用户不得被迫从空白状态重新开始

#### 场景:游客刷新当前浏览器页面
- **当** 游客在同一浏览器中刷新页面
- **那么** 系统必须恢复该游客最近一次活动会话
- **那么** 已有聊天历史和当前结果必须继续可见

### 需求:用户只能访问自己的会话资产
系统必须确保会话和回合查询受到归属校验，用户不得访问其他用户的会话；游客也不得访问不属于当前游客标识的会话。

#### 场景:用户尝试访问其他人的会话
- **当** 用户请求一个不属于自己的会话或回合
- **那么** 系统必须拒绝访问
- **那么** 系统不得返回该会话的消息、约束或结果内容
