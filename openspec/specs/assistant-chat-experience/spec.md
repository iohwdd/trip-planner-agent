## 需求
### 需求:系统必须提供独立的智能助手入口
系统必须提供与旅行规划工作台分离的智能助手页，禁止再将聊天界面作为流程化规划工作台的默认承载面。

#### 场景:用户进入首页
- **当** 用户打开应用主入口
- **那么** 系统必须将智能助手作为一个独立入口呈现，并明确其职责是通用问答与对话支持

### 需求:智能助手必须使用独立的会话与消息模型
系统必须以独立的助手会话和消息记录承载对话上下文，禁止继续复用旅行规划专用的会话/回合/结果模型。

#### 场景:用户发起一轮助手对话
- **当** 用户在智能助手中发送消息
- **那么** 后端必须将其记录到助手域的会话和消息中，并返回一条助手回复，而不是旅行规划结果对象

### 需求:智能助手回复必须由系统提示词约束
系统必须通过一个明确的系统提示词约束助手回复行为，并为后续扩展联网搜索和 RAG 保留独立演进空间。

#### 场景:用户提出非旅行规划问题
- **当** 用户在智能助手中提出普通问题
- **那么** 系统必须返回普通问答式回复，而不是尝试生成旅行规划结果

### 需求:智能助手必须在知识库可用时执行检索增强
系统必须在知识库中存在可用文档时，对智能助手消息执行检索增强生成；当知识库为空或检索失败时，系统必须回退为普通助手回答，而不得中断现有会话流程。

#### 场景:知识库命中用户问题
- **当** 用户在智能助手中发送一条与知识库内容相关的消息
- **那么** 系统必须先执行向量召回与重排
- **那么** 系统必须将命中的知识块作为上下文注入助手生成链路

#### 场景:知识库为空或无有效命中
- **当** 默认知识库没有可用文档，或当前问题未检索到足够的相关内容
- **那么** 系统必须继续返回普通助手回复
- **那么** 系统不得因为知识库不可用而导致消息请求失败

### 需求:智能助手流式协议必须兼容知识库增强
系统必须保证知识库增强后的助手回复仍然沿用现有的助手消息接口和流式事件协议，禁止因为引入 RAG 而更改前端消费方式。

#### 场景:用户以流式模式发送助手消息
- **当** 客户端调用现有的助手流式消息接口
- **那么** 服务端必须仍然返回 `message.delta` 和 `message.complete` 事件
- **那么** 前端不得因为知识库增强而需要切换到新的接口协议
