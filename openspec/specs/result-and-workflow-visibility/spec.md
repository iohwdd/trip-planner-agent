## 需求
### 需求:结果区必须以用户决策信息为主线组织内容
系统必须优先展示用户最关心的路线摘要、当前状态、关键约束和下一步动作，再分层展示路线细节、交通衔接、预算、替代方案、修订说明和数据来源等辅助模块。

#### 场景: 展示已生成路线结果
- **当** 系统已经产出可消费的路线结果
- **那么** 结果区必须先展示摘要级内容，再展示各专题模块，且模块顺序必须稳定可预期

### 需求:结果模块必须在各种状态下保持样式稳定
系统必须对空态、加载态、澄清态、完成态、降级态和失败态提供一致的样式规则，禁止出现卡片错位、间距异常、信息重叠或模块层级混乱。

#### 场景: 显示澄清态结果
- **当** 当前回合返回澄清问题而非最终路线
- **那么** 结果区必须以澄清提示和待补充信息为主，不得错误渲染成完整路线卡片布局

#### 场景: 显示降级态结果
- **当** 某些数据源或路径规划降级但仍可返回结果
- **那么** 结果区必须清晰标示降级状态、影响范围和建议动作，同时保持整体布局稳定

### 需求:执行过程必须清晰表达当前步骤与已完成阶段
系统必须将执行过程展示为可理解的阶段视图，使用户能够直接看出当前执行到了哪一步、哪些步骤已完成、哪些步骤存在异常，以及这些步骤与当前结果之间的关系。

#### 场景: 回合执行中查看过程
- **当** 用户在回合运行过程中查看执行过程区域
- **那么** 界面必须高亮当前步骤，并同时展示已完成步骤和可用的阶段性结果
