# admin-knowledge-base-management 规范

## 目的
定义管理员知识库的访问边界、文档管理流程与分层持久化约束，使智能助手的 RAG 数据源具备稳定的后台维护能力。

## 需求
### 需求:系统必须提供仅管理员可访问的知识库管理入口
系统必须在产品主导航中提供与智能助手、工作台同级的知识库入口，并且该入口只对 `is_staff=true` 的管理员可见且可访问。

#### 场景:管理员进入应用
- **当** 已登录且 `is_staff=true` 的用户恢复身份状态
- **那么** 前端必须在主导航中展示知识库入口
- **那么** 用户进入知识库页面时系统必须允许访问文档管理能力

#### 场景:普通用户进入应用
- **当** 未登录用户或 `is_staff=false` 的已登录用户恢复身份状态
- **那么** 前端不得展示知识库入口
- **那么** 用户直接访问知识库接口或页面时系统必须返回权限受限结果

### 需求:管理员必须能够管理知识库文档
系统必须允许管理员上传、查看、删除、单文档重试和重建知识库文档索引，并且文档类型必须限制为 `md`、`txt` 和 `pdf`。

#### 场景:管理员上传支持的文档
- **当** 管理员上传一个 `md`、`txt` 或带文本层的 `pdf` 文件
- **那么** 系统必须保存该文件的元数据与处理状态
- **那么** 系统必须开始解析、切分和索引该文档

#### 场景:管理员上传不支持的文档
- **当** 管理员上传 `md`、`txt` 和 `pdf` 之外的文件
- **那么** 系统必须拒绝该请求
- **那么** 系统必须返回可直接消费的错误信息

#### 场景:管理员上传纯扫描 PDF
- **当** 管理员上传不含文本层的图片型 `pdf`
- **那么** 系统必须保留失败状态与错误信息
- **那么** 系统必须明确告知当前版本暂不支持该类 PDF 的索引

#### 场景:管理员删除文档
- **当** 管理员删除一个已存在的知识库文档
- **那么** 系统必须删除该文档的元数据记录
- **那么** 系统必须清理对应的对象存储文件与 Chroma 向量条目

#### 场景:管理员重试失败文档
- **当** 管理员对一个失败状态的知识库文档发起重试
- **那么** 系统必须基于对象存储中的原始文件重新执行解析、切分和索引流程
- **那么** 系统必须更新该文档最新的处理状态、错误信息与 chunk 统计

#### 场景:管理员重建索引
- **当** 管理员对默认知识库发起重建索引操作
- **那么** 系统必须基于对象存储中的原始文件重新执行解析、切分和索引流程
- **那么** 系统必须更新各文档最新的处理状态与 chunk 统计

### 需求:系统必须分层持久化知识库内容
系统必须将知识库原始文件、管理元数据和向量索引分层持久化，禁止只依赖单一存储承载全部知识库数据。

#### 场景:文档上传成功
- **当** 管理员成功上传知识库文档
- **那么** 系统必须将原始文件保存到 MinIO `trip-assistant` bucket
- **那么** 系统必须在关系库中保存对象键、文档状态和必要的管理字段
- **那么** 系统必须将解析后的 chunk 与 embedding 持久化到 ChromaDB

#### 场景:索引失败
- **当** 文档解析、向量化或写入 Chroma 过程中发生错误
- **那么** 系统必须保留该文档的失败状态和错误信息
- **那么** 系统不得错误地将该文档标记为可用于检索
