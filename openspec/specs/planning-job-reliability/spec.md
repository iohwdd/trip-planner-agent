## 需求
### 需求:规划任务必须以持久化队列方式异步执行
系统必须将旅行规划请求和聊天回合写入可持久化任务队列，由独立执行器异步处理，而不得依赖 API 进程内线程来完成任务。

#### 场景:用户提交新的聊天回合
- **当** 用户发送一条新的聊天消息或提交一次快捷表单
- **那么** 系统必须创建一个可查询的任务记录并返回排队状态
- **那么** 任务执行必须由独立 worker 处理，而不是绑定当前 HTTP 请求线程

### 需求:任务状态必须在服务重启后可恢复
系统必须在数据库中持久化任务状态、回合状态和结果快照，确保服务进程重启后历史数据和待处理任务不会整体丢失。

#### 场景:服务在任务创建后重启
- **当** 某个规划任务已经入队或正在执行时服务发生重启
- **那么** 系统必须保留该任务和对应回合的状态记录
- **那么** 重启后的执行器必须能够识别并继续处理或明确标记异常任务

### 需求:系统必须明确记录排队、运行、完成、失败和超时状态
系统必须为每个规划任务和对应回合提供清晰的状态机，并在失败或超时时向前端返回明确错误信息。

#### 场景:规划任务执行超时
- **当** 某个规划任务超过设定的超时时间仍未完成
- **那么** 系统必须把该任务标记为超时失败或等价失败状态
- **那么** 前端查询该回合时必须能够看到明确的失败原因

#### 场景:外部数据源或模型调用失败
- **当** 任务在执行过程中出现模型调用失败或数据源异常
- **那么** 系统必须记录错误原因和失败阶段
- **那么** 前端必须能够展示失败或降级状态，而不是无限等待

### 需求:系统必须记录基础运营和性能观测数据
系统必须记录会话数、回合数、方案保存数、澄清轮次、游客转登录事件，以及模型调用耗时、数据源调用耗时和失败率等基础指标。

#### 场景:一次规划任务成功完成
- **当** 一个聊天回合或单次规划任务完成
- **那么** 系统必须记录该任务的总耗时、关键步骤耗时和结果状态
- **那么** 系统必须允许后续统计该任务是否经历了澄清、降级或方案保存

#### 场景:游客完成登录迁移
- **当** 游客在已有会话后完成注册或登录
- **那么** 系统必须记录一次游客到注册用户的转化事件
- **那么** 后续统计必须能够把该事件与原始游客会话关联起来
