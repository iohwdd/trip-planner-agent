## 需求
### 需求:会话页必须突出每条会话的继续上下文
系统必须在会话页中展示每条会话当前停留原因、最近状态和继续价值，禁止仅用标题、摘要和删除按钮表达历史会话。

#### 场景:浏览历史会话
- **当** 用户进入会话页
- **那么** 每条会话必须清楚说明它是等待澄清、已有草案、已完成还是失败中断，以及继续编辑会从什么上下文恢复

### 需求:方案页必须突出版本语义与继续修订路径
系统必须在方案页和方案详情中明确表达草案、最终版、来源上下文和继续修订关系，帮助用户判断哪一版最适合继续推进。

#### 场景:浏览历史方案列表
- **当** 用户查看方案页
- **那么** 用户必须可以快速区分草案与最终版，并理解继续修订后会创建新的可编辑会话

#### 场景:查看单个方案详情
- **当** 用户打开某个历史方案
- **那么** 页面必须展示该版本的关键约束快照、结果快照和继续修订入口，而不是仅充当只读归档页

### 需求:个人中心必须服务于旅行资产连续性
系统必须将个人中心定义为身份与旅行资产状态的承载面，禁止让其主要价值退化为通用账户管理页。

#### 场景:已登录用户进入个人中心
- **当** 用户查看个人中心
- **那么** 页面必须优先展示身份状态、资产概览和与继续规划相关的能力，而不是把账户设置放在唯一核心位置

#### 场景:游客进入个人中心
- **当** 游客查看个人中心
- **那么** 页面必须明确说明游客当前可恢复的资产和登录迁移收益
