## 需求
### 需求:对话式路由规划澄清
系统必须支持用户通过多轮对话逐步描述目的地、预算、天数、交通偏好、必去点位和节奏要求，并在信息不足时主动发起澄清。

#### 场景:用户输入仍然模糊
- **当** 用户只提供“想在上海边走边逛一天，不要太赶”这类不足以直接生成路线的描述
- **那么** 系统必须返回结构化澄清问题
- **那么** 系统必须保留当前已识别出的目的地和节奏偏好

### 需求:对话中保留已确认约束
系统必须在同一会话中保留已确认的出行约束，并在后续回合继续沿用这些约束，直到用户明确修改。

#### 场景:用户分多轮补充条件
- **当** 用户先说明“预算 3000、两天”并在下一轮追加“主要坐地铁，想去博物馆和外滩”
- **那么** 系统在生成路线草案时必须同时使用前后两轮中已确认的条件
- **那么** 系统不得要求用户重复输入已经确认的信息

### 需求:支持基于草案的路线修订
系统必须允许用户针对已有路线草案提出修订请求，并据此生成更新后的路线方案。

#### 场景:用户要求替换站点
- **当** 用户在收到路线草案后提出“不要外滩，换成苏州河沿线”
- **那么** 系统必须保留未被否定的既有约束
- **那么** 系统必须返回更新后的路线草案或新的澄清问题
