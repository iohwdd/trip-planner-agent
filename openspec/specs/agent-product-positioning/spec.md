## 需求
### 需求:产品必须统一表达为个人旅行规划 Agent 工作台
系统必须在壳层、主工作台、资产页和身份入口中一致表达“通过对话持续打磨旅行决策，并将结果沉淀为可继续编辑的资产”的产品定位，禁止在关键入口中把产品表述成一次性攻略生成器或泛化聊天工具。

#### 场景:首次进入产品
- **当** 用户首次进入应用首页或工作台
- **那么** 界面必须优先说明聊天式规划、路线沉淀和持续修订之间的关系

#### 场景:从资产页回到主工作流
- **当** 用户查看会话页、方案页或个人中心
- **那么** 页面必须明确说明这些资产与继续规划之间的关系，而不是只呈现管理动作

### 需求:一级导航必须围绕主任务组织
系统必须将一级导航组织为围绕旅行规划主任务的入口，禁止让与主任务无关的账户管理能力在默认层级中与工作台并列争夺注意力。

#### 场景:浏览一级导航
- **当** 用户展开应用导航
- **那么** 用户必须可以快速区分哪些入口用于继续规划，哪些能力只是辅助支撑

### 需求:游客态与登录态的产品承诺必须清晰
系统必须在游客态和登录态下清晰说明各自可以做什么、不能做什么，以及登录后会获得什么连续性收益。

#### 场景:游客查看顶部或身份入口
- **当** 用户处于游客模式
- **那么** 系统必须明确提示游客可以体验聊天规划，并说明登录后可保存与迁移资产

#### 场景:已登录用户查看顶部或身份入口
- **当** 用户已经登录
- **那么** 系统必须明确说明当前账户已具备管理会话、保存方案和继续修订等能力
